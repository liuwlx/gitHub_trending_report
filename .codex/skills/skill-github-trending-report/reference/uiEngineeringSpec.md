# GitHub Trending Report UI Engineering Spec

## 1. 文档定位

这份规格不再用于“重新设计一张日报页面”，而是用于说明如何把 `2026-03-16` 那份已验收成品页，当作固定 UI 工程底板来使用。

当前 skill 的页面工程基线是：

- 固定模板：`reference/fixedBaseTemplate_2026-03-16.html`
- 原始来源：`github/github_trending_report_2026-03-16.html`
- 用户硬约束：`reference/user-rules.md`

换句话说，这份文档负责解释“该怎么忠实复用底板”，而不是鼓励重新发明布局。

## 2. 使用原则

### 2.1 优先级

执行顺序必须遵循：

1. `reference/user-rules.md`
2. `reference/fixedBaseTemplate_2026-03-16.html`
3. `reference/uiEngineeringSpec.md`
4. 本次任务生成的本地 Markdown 底稿

### 2.2 当前工程目标

最终 HTML 必须同时满足三件事：

1. 看起来仍然是 `03-16` 那张日报，而不是一张新页面。
2. 内容改成当天真实数据和当天文案。
3. 明细区的阅读效率、层级和交互保持不变。

### 2.3 生成方式

- 必须先生成本地 Markdown 底稿
- 再基于固定底板写最终 HTML
- 最终 HTML 由当前 AI 直接编写或改写
- 不允许交给 Python 脚本、本地渲染器或固定模板程序代写

## 3. 固定页面骨架

除非用户明确批准改动，否则页面顺序固定如下：

1. 右上角主题切换按钮
2. Header / Hero
3. 4 张统计卡
4. 今日洞察
5. 今日热门 Top 12
6. 编程语言分布
7. GitHub Explore 精选
8. Footer

这里的“固定”不仅指区块顺序，也指阅读节奏：

- 首屏先看今天的总盘面
- 中段先扫榜单，再按需展开
- 下段看语言结构与 Explore 补充
- 最后看来源与生成说明

## 4. 固定视觉与交互契约

### 4.1 必须保留的视觉语言

以下风格是底板的一部分，默认不得改掉：

- GitHub 风格深色主题变量
- 明暗主题切换逻辑
- 居中单栏阅读布局
- 轻边框、轻阴影、信息密度高但不拥挤的卡片节奏
- Top 3 的克制高亮
- Explore 的紧凑列表样式

### 4.2 必须保留的 CSS token

以下变量体系默认视为固定约定：

- `--bg`
- `--surface`
- `--surface2`
- `--border`
- `--text`
- `--text-muted`
- `--blue`
- `--green`
- `--orange`
- `--purple`
- `--red`
- `--yellow`

可以替换数据和内容，不可以重写为另一套主题系统。

### 4.3 必须保留的交互

底板当前只有两个交互，默认都要保留：

- `toggleTheme()`：负责明暗主题切换
- `toggleDetail(btn)`：负责 Top 项目展开与收起

展开详情按钮文案仍然应维持：

- 收起前：`▼ 展开详情`
- 展开后：`▲ 收起详情`

## 5. 内容映射规范

HTML 必须从本地 Markdown 底稿映射，而不是凭空生成。

### 5.1 Header / Hero 映射

Markdown 中以下信息应映射到首屏：

- Hero 主标题建议 -> `header h1`
- Hero 副标题建议 -> `.subtitle`
- Report Date / Sources -> `.date-badge`

### 5.2 Stats 映射

Markdown 的 4 个核心指标固定映射到 4 张统计卡：

1. Trending 项目数
2. 今日累计新增 Stars
3. 编程语言数
4. AI 相关项目数

不要增减卡片数量，也不要改成其他指标组合。

### 5.3 Insights 映射

Markdown 的 `Editorial Insights` 应映射为 3 到 4 条编辑观察。

每条保持：

- 一个醒目的标题
- 一段完整、可读、不是口号的正文

### 5.4 Top 12 封面态映射

每张项目卡的封面态必须至少包含：

- 排名
- 仓库名与链接
- 一句简介或定位
- Stars / Forks / 语言 / License 等 meta
- topics badge
- 展开按钮

### 5.5 Top 12 展开态映射

每张项目卡的展开态必须严格保留以下顺序：

1. `📖 项目摘要`
2. `✨ 核心特性`
3. `🔧 技术栈`
4. `🎯 适用场景`
5. `💬 一句话推荐`

映射关系如下：

- `项目摘要` -> 第一段完整说明
- `核心特性` -> 3 条左右 feature list
- `技术栈` -> `tech-tags`
- `适用场景` -> 一段场景判断
- `一句话推荐` -> `recommend-quote`

### 5.6 Language Distribution 映射

语言分布应继续使用横向比例条：

- 名称
- 条形进度
- 数量 + 百分比

不要改成饼图、徽章堆叠或卡片矩阵。

### 5.7 Explore 映射

Explore 保持紧凑编号列表：

- 编号
- 链接标题
- 一小段 meta 或简短原因

如果 Explore 信息不足，也应保留这个区域，并明确写降级说明。

## 6. 文案密度规范

### 6.1 文案优先级

HTML 里的中文文案优先级如下：

1. 读者是否能快速理解项目到底是什么
2. 读者是否能判断自己该不该点开
3. 读者是否知道第一次该看哪里

### 6.2 不允许的写法

不允许把下面这些内部观察直接交给用户：

- README 很完整
- topics 很聚焦
- examples 很多
- docs 很齐全
- 生态活跃

这些可以作为写作依据，但不能直接充当用户文案。

### 6.3 允许的压缩边界

如果某段文案过长，只允许做轻量排版收束：

- 删重复表达
- 调顺句子
- 把一句过长的话拆成两句

不允许把原本已经解释清楚的内容压成一句模糊总结。

## 7. 响应式与实现约束

- 继续保留固定底板当前的移动端断点策略
- 手机端优先保证项目名、meta、展开按钮不打架
- 详情区内容宁可换行，也不要删掉核心句子
- 不要引入额外框架、复杂脚本或大型交互组件

## 8. 验收口径

最终交付前，至少确认：

- HTML 明显继承自 `reference/fixedBaseTemplate_2026-03-16.html`
- 页面区块顺序、类名、展开交互仍与底板一致
- Markdown 底稿内容被完整映射，没有被二次糊弄式改写
- `项目摘要 / 核心特性 / 技术栈 / 适用场景 / 一句话推荐` 五段仍然完整存在
- Explore 仍然是列表，不是新设计
- 如果某些项目理解仍然有限，页面已明确写出诚实 caveat
